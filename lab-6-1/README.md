
# OSCAL Compliance Mapping & Evidence Directory

This directory contains the machine-readable Open Security Controls Assessment Language (OSCAL) models for our foundational infrastructure components. By defining our compliance posture in OSCAL, we transform traditional manual audits into an automated **graph traversal** that links our infrastructure declarations directly to active, cryptographic evidence.

## Component to Module Mapping

Our OSCAL architecture maps software primitives directly to the infrastructure patterns that implement them.

| OSCAL Component File | Managed Primitive | Implementation Module Reference |
| --- | --- | --- |
| `component-definitions/compliant-s3-v1/component-definition.json` | `compliant-s3` | `reference/lab-2-3/` (Reusable S3 + Logging Terraform pattern) |

### Implemented Controls (NIST SP 800-53 Rev 5)

The `compliant-s3` component definition declares exact implementation requirements for the following security controls:

* **SC-28 (Protection of Information at Rest):** Enforced via hardcoded AES-256 server-side encryption configurations (`aws_s3_bucket_server_side_encryption_configuration.primary`).
* **AC-3 (Access Enforcement):** Public access explicitly blocked on all buckets with all four protection flags forced to true via `aws_s3_bucket_public_access_block`.
* **AU-3 (Audit Record Content):** Automated server access logging routed to a secondary, isolated log bucket via `aws_s3_bucket_logging.primary`.
* **CM-6 (Configuration Settings):** Strict enforcement of mandatory organizational tags (`Project`, `Environment`, `ManagedBy`, `ComplianceScope`) via default provider configurations.

## The Evidence Chain Architecture

Rather than relying on static documentation or human assertions, each control implementation requirement points directly to automated, verifiable evidence.

```
OSCAL Component Definition
  └── implemented-requirements (e.g., sc-28)
        └── links [rel="evidence"]
              └── href: s3://cgep-lab-grc-evidence-vault-[ID]/runs/{RUN_ID}/

```

### Where the Evidence Lives

All proof of compliance is securely shipped to our immutable GRC vault:

* **S3 Evidence Vault Bucket:** `cgep-lab-grc-evidence-vault-[ID]`
* **Storage Path Convention:** `runs/${RUN_ID}/`

Each run path contains a complete cryptographic attestation bundle generated by the deployment pipeline:

1. **`evidence-${RUN_ID}-${HASH}.tar.gz`**: The primary payload containing the actual `plan.json` configurations and automated Conftest evaluation results.
2. **`evidence-${RUN_ID}-${HASH}.tar.gz.sha256`**: The SHA-256 hash verifying raw file integrity.
3. **`evidence-${RUN_ID}-${HASH}.tar.gz.sig.bundle`**: The Sigstore/Cosign bundle establishing identity authenticity and OIDC-issued timestamp verification via GitHub Actions.
4. **`receipt.json`**: The pipeline execution metadata statement.

## How to Verify the Graph Traversal

An automated pipeline or independent assessor can programmatically verify the intact chain of assurance without system administrator intervention.

### 1. Validate the OSCAL Layer

Ensure the compliance model complies fully with the strict NIST OSCAL schema definitions using `compliance-trestle`:

```bash
# Validate the Component Definition
trestle validate -f component-definitions/compliant-s3-v1/component-definition.json

# Validate the Control Profile Selection
trestle validate -f profiles/cge-p-minimum/profile.json

```

### 2. Verify the Vault Artifacts

Extract the `href` target from the control you wish to audit, isolate the corresponding pipeline `RUN_ID`, and execute the automated preservation script to verify integrity, authenticity, and Object Lock status:

```bash
# Set your target lab vault
export EVIDENCE_VAULT="cgep-lab-grc-evidence-vault-[ID]"

# Execute the verification routine against the run ID
bash scripts/verify-evidence.sh <RUN_ID>

```

If the verification succeeds, the script will output `CHAIN INTACT`, confirming that the configurations running live in our cloud match the exact requirements mapped in the accompanying OSCAL definitions.
